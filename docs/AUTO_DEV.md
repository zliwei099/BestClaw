## 自动迭代流程

```
1. 检查开发路线图
      ↓
2. 选择下一个任务
      ↓
3. 技术方案设计
      ↓
4. 代码实现
      ↓
5. 本地测试
      ↓
6. 提交代码 (git commit)
      ↓
7. 推送至 GitHub
      ↓
8. 更新开发日志
      ↓
9. 规划下一个任务
      ↓
10. 等待下一轮迭代
```

## 迭代触发机制

### 方式1: Cron 定时任务 (推荐)
```bash
# 每小时检查一次
0 * * * * cd ~/dev/openclaw/BestClaw && openclaw run auto-dev

# 或每4小时执行一次开发迭代
0 */4 * * * cd ~/dev/openclaw/BestClaw && openclaw run dev-iteration
```

### 方式2: Heartbeat 检查
在 HEARTBEAT.md 中添加检查项，每次心跳时评估是否需要开发。

### 方式3: 事件驱动
- GitHub Webhook 触发
- 代码提交后自动规划下一个任务
- 检测到 Bug 后自动修复

## 开发规范

### 代码提交规范
```
类型: 简短描述

详细说明

- 影响范围: xxx
- 测试情况: xxx
```

类型包括:
- `feat:` 新功能
- `fix:` 修复
- `docs:` 文档
- `refactor:` 重构
- `test:` 测试
- `chore:` 构建/工具

## 扩展指令

### 用户干预
用户可以通过指令控制开发:
- `@bestclaw dev:pause` - 暂停自动开发
- `@bestclaw dev:resume` - 恢复自动开发
- `@bestclaw dev:priority <task>` - 设置优先级任务
- `@bestclaw dev:status` - 查看开发状态

### 智能决策
系统自主决定:
- 选择下一个开发任务
- 评估技术方案
- 决定重构时机
- 优化代码结构
